## 배경

- 현재 케이스 스터디가 아코디언 카드 안에 텍스트만으로 구성되어 있어, "왜, 어떻게 했는가"를 충분히 전달하기 어려움
- 채용 담당자가 특정 케이스를 팀에 공유하고 싶어도 개별 URL이 없어 불가능
- 이력서의 핵심 차별화 요소인 케이스 스터디가 현재 포맷의 한계로 제 역할을 못 하고 있음

## 사용자 스토리

### 스토리 1: 케이스 스터디 상세 읽기
- **사용자**: 채용 담당자
- **행동**: 목록에서 관심 있는 케이스를 클릭해 상세 페이지로 이동한다
- **결과**: 이미지, 코드 블록이 포함된 풍부한 콘텐츠를 블로그처럼 읽을 수 있다
- **우선순위**: P0 (필수)

### 스토리 2: 특정 케이스 공유
- **사용자**: 채용 담당자
- **행동**: 인상 깊은 케이스의 URL을 슬랙/이메일로 팀에 공유한다
- **결과**: 받은 사람이 해당 케이스 페이지로 바로 접근할 수 있다
- **우선순위**: P0 (필수)

### 스토리 3: 검색엔진에서 케이스 발견
- **사용자**: 잠재 채용 담당자
- **행동**: "인스타그램 DM 자동화 설계" 같은 키워드로 검색한다
- **결과**: 해당 케이스 스터디 페이지가 검색 결과에 노출된다
- **우선순위**: P1 (중요)

### 스토리 4: 목록에서 케이스 탐색
- **사용자**: 방문자
- **행동**: 사이드바에서 케이스 스터디를 클릭해 목록 페이지를 본다
- **결과**: 각 케이스의 제목, 요약, 태그를 보고 관심 있는 것을 선택할 수 있다
- **우선순위**: P0 (필수)

## 성공 지표

| 지표 | 목표 | 측정 방법 |
|------|------|-----------|
| 개별 URL | 3개 케이스 각각 고유 URL | 수동 확인 |
| SSG | 상세 페이지가 정적 생성 | `pnpm build` 출력 확인 |
| SEO | title, description, OG 메타태그 | 수동 확인 |
| MDX 렌더링 | 이미지, 코드 블록, 소제목 정상 표시 | 수동 확인 |
| 기존 이력서 | 이력서 탭 정상 동작 | 수동 확인 |

## 범위

### 포함
- MDX 콘텐츠 3개 (기존 데이터 이관 + 마크다운 구조화)
- 목록 페이지 (`/case-study`)
- 상세 페이지 (`/case-study/[slug]`)
- 사이드바 네비게이션 연결
- SEO 메타태그

### 제외
- 기술 블로그 시스템
- 댓글, 태그 필터, 검색, RSS
- 이미지 제작 (텍스트 콘텐츠만 먼저)
